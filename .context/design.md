## 状态管理分工与边界

### React Hook Form（表单状态管理）

- **专注职责**：

  - 处理所有表单相关状态（字段值、校验状态、错误信息、提交状态）
  - 管理表单验证逻辑和错误展示
  - 处理表单字段之间的依赖关系
  - 优化表单性能（避免不必要的重渲染）

- **使用时机**：

  - 创建任何用户输入表单（登录、注册、数据编辑等）
  - 需要复杂字段验证的场景
  - 表单字段之间存在依赖关系
  - 需要高性能表单体验

- **禁止模式**：
  - ❌ 不要在 React Hook Form 管理的表单中混用 `useState`
  - ❌ 不要将表单状态存储在 Zustand 中（除非确实需要跨组件共享表单状态）
  - ❌ 不要将表单提交逻辑与状态更新混在一起

### SWR（数据获取与缓存管理）

- **专注职责**：

  - 所有远程数据请求（API、Supabase 查询）
  - 数据缓存与自动重新验证
  - 处理加载和错误状态
  - 支持数据预取和条件请求
  - 实现乐观更新和实时数据同步

- **使用时机**：

  - 从服务器获取任何数据
  - 需要缓存服务器响应
  - 需要自动刷新数据（如窗口聚焦时）
  - 实现数据依赖链（一个请求依赖另一个请求的结果）
  - 需要乐观更新 UI（先更新界面再等待服务器确认）

- **禁止模式**：
  - ❌ 不要在 SWR 中存储纯客户端状态
  - ❌ 不要用 SWR 管理表单状态
  - ❌ 不要在 SWR 和 Zustand 之间复制完整的原始数据

### Zustand（全局状态协调）

- **专注职责**：

  - 管理全局 UI 状态（侧边栏状态、主题、语言等）
  - 存储关键用户数据的派生状态（如权限标志、首选项）
  - 管理跨组件共享的应用状态
  - 持久化用户偏好设置

- **使用时机**：

  - 多个不相关组件需要共享状态
  - 需要持久化用户设置（主题、语言等）
  - 从 SWR 数据创建派生状态（如权限标志）
  - 管理应用级工作流状态（如多步骤向导）

- **禁止模式**：
  - ❌ 不要在 Zustand 中发起 API 请求（由 SWR 处理）
  - ❌ 不要在 Zustand 中存储完整的原始 API 响应
  - ❌ 不要将仅在单个组件中使用的状态放入 Zustand
  - ❌ 不要使用 Zustand 管理表单状态（除非确实需要跨组件共享）

## 三者协作最佳实践

### 数据流向与职责分离

1. **远程数据流**：

   - SWR 负责获取和缓存服务器数据
   - 组件直接消费 SWR 数据进行渲染
   - 需要时，将 SWR 数据的派生状态存入 Zustand（如角色权限标志）

2. **表单数据流**：

   - React Hook Form 管理所有表单状态和验证
   - 表单提交时，通过 SWR 的 mutate 方法发送数据
   - 成功后，可使用 SWR 的 mutate 乐观更新缓存

3. **UI 状态流**：
   - Zustand 管理全局 UI 状态和用户偏好
   - 组件通过 Zustand hooks 直接访问全局状态
   - 用户偏好通过 Zustand persist 中间件持久化

## 状态管理黄金法则

1. **表单状态**：始终使用 React Hook Form，不混用其他状态管理方案
2. **远程数据**：始终通过 SWR 获取、缓存和同步
3. **跨组件状态**：仅将真正需要共享的状态放入 Zustand
4. **保持单一职责**：每个状态只存在于一个地方，避免冗余复制

## 一句话总结

**SWR 负责所有远程数据获取与缓存，React Hook Form 专注处理表单状态与验证，Zustand 只管理必要的全局/跨组件状态 — 遵循这一原则，代码将更易维护、更加高效、更少出错。**
