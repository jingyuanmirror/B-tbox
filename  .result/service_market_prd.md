# 百宝箱企业版服务市场PRD设计文档

## 1. 项目背景

### 1.1 平台简介
百宝箱是一款支持一站式AI原生应用的开发平台，企业版基于业务专属训练模型与行业知识深度融合，为出行、景区和生活服务等行业提供智能化解决方案。平台集Agent构建、开放生态、模型中心等能力为一体，助力企业快速构建具备业务理解和场景适应的智能应用。

### 1.2 升级背景
原有插件市场仅支持单一的插件类型服务，无法满足日益多样化的AI应用构建需求。为了更好地服务企业客户，需要将插件市场升级为更综合的服务市场，支持多种服务类型，提升平台的服务能力和生态丰富度。

## 2. 项目目标

### 2.1 业务目标
- 扩展服务类型覆盖范围，从单一插件扩展到多种服务形态
- 提升用户服务获取效率，支持更精准的服务发现
- 构建完善的服务生态，促进服务提供者和使用者的双向成长
- 增强平台竞争力，提供差异化的服务体验

### 2.2 用户目标
- 服务使用者：更便捷地找到合适的服务，提升应用构建效率
- 服务提供者：更好的服务展示和分发平台，获得更多曝光和使用

## 3. 需求分析

### 3.1 现状分析
**原插件市场特点：**
- 服务类型单一：仅支持插件类型
- 使用方式固定：插件订阅后引用到智能体
- 功能相对简单：基本的搜索和分类功能

**升级需求：**
- 支持多种服务类型：插件、MCP、服务流、代码服务
- 差异化使用方式：订阅引用 vs 购买实例化
- 完善的服务管理：上架、权限控制、资源管理

### 3.2 用户角色分析
1. **服务使用者**：企业开发者、智能体构建者
2. **服务提供者**：插件开发者、工作流创建者、代码服务开发者
3. **平台管理者**：服务审核、生态运营

## 4. 产品架构

### 4.1 服务类型定义

#### 4.1.1 插件（Plugin）
- **定义**：可复用的功能组件，提供特定能力
- **使用方式**：订阅后可被引用到不同智能体中
- **特点**：轻量级、即插即用、多智能体共享

#### 4.1.2 MCP（Model Context Protocol）
- **定义**：模型上下文协议服务，提供结构化的上下文交互能力
- **使用方式**：订阅后可被引用到不同智能体中
- **特点**：协议标准化、上下文感知、跨模型兼容

#### 4.1.3 服务流（Service Flow）
- **定义**：基于工作流构建的复杂服务模板
- **使用方式**：购买后实例化到本空间，成为用户自有资产
- **特点**：可定制化、支持二次编辑、完整业务流程

#### 4.1.4 代码服务（Code Service）
- **定义**：基于代码构建的自定义服务模板
- **使用方式**：购买后实例化到本空间，成为用户自有资产
- **特点**：高度定制、代码级控制、专业开发

### 4.2 功能模块设计

#### 4.2.1 服务发现模块
**功能分类筛选**
- 搜索类：文档搜索、网络搜索、数据查询等
- 视频类：视频生成、视频分析、视频编辑等
- 图片类：图像生成、图像识别、图像处理等
- 新闻类：新闻获取、内容分析、信息聚合等
- 数据类：数据分析、数据可视化、数据处理等
- 通信类：邮件发送、消息推送、API调用等
- 工具类：文件处理、格式转换、计算服务等

**搜索功能**
- 服务名称精确搜索
- 服务描述关键词搜索
- 标签组合搜索
- 智能推荐搜索

**排序机制**
- 热度排序：基于使用次数、用户评分
- 时间排序：上架时间、更新时间
- 相关度排序：与搜索关键词的匹配程度
- 综合排序：综合多个维度的权重计算

#### 4.2.2 服务详情展示模块
**基础信息**
- 服务名称、版本号、开发者信息
- 服务类型、功能分类标签
- 服务描述、使用说明
- 上架时间、更新时间

**使用统计**
- 使用次数、用户评分
- 用户评价和反馈
- 兼容性信息

**操作区域**
- 订阅/购买按钮
- 预览/试用功能
- 收藏、分享功能

#### 4.2.3 服务管理模块
**我的服务**
- 已订阅服务列表
- 已购买服务实例
- 使用历史记录
- 服务更新通知

**服务上架**
- 服务信息填写
- 资源权限检查
- 服务测试验证
- 上架申请流程

#### 5.2.2 操作流程设计

**插件/MCP订阅流程**
1. 用户点击"订阅"按钮
2. 确认订阅弹窗（显示权限说明）
3. 订阅成功，可在智能体中引用
4. 订阅后可在"我的服务"中管理

**服务流/代码服务购买流程**
1. 用户点击"购买"按钮
2. 权限检查（检查服务中的资源权限）
3. 如有私有资源，提示无法购买
4. 购买确认（价格、权限说明）
5. 购买成功，自动实例化到用户空间
6. 用户可对实例进行二次编辑

### 6.3 数据库设计

#### 6.3.1 服务表设计
```sql
CREATE TABLE services (
  id VARCHAR(64) PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  type ENUM('plugin', 'mcp', 'workflow', 'code') NOT NULL,
  category VARCHAR(100),
  description TEXT,
  developer_id VARCHAR(64),
  version VARCHAR(20),
  status ENUM('draft', 'reviewing', 'published', 'rejected') DEFAULT 'draft',
  download_count INT DEFAULT 0,
  rating_avg DECIMAL(3,2) DEFAULT 0.00,
  rating_count INT DEFAULT 0,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  published_at TIMESTAMP NULL,
  
  INDEX idx_type_category (type, category),
  INDEX idx_status_published (status, published_at),
  INDEX idx_developer (developer_id)
);
```

#### 6.3.2 用户服务关系表
```sql
CREATE TABLE user_services (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id VARCHAR(64) NOT NULL,
  service_id VARCHAR(64) NOT NULL,
  relation_type ENUM('subscribed', 'purchased') NOT NULL,
  instance_id VARCHAR(64) NULL, -- 购买类型的实例ID
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  UNIQUE KEY uk_user_service (user_id, service_id),
  INDEX idx_user (user_id),
  INDEX idx_service (service_id)
);
```

## 7. 用户体验设计

### 7.1 交互设计原则
- **一致性**：不同服务类型保持一致的操作体验
- **清晰性**：明确区分订阅和购买的不同操作
- **反馈性**：及时的操作反馈和状态提示
- **容错性**：友好的错误处理和引导

### 7.2 关键路径优化
1. **服务发现路径**：首页 → 分类筛选 → 服务详情 → 订阅/购买
2. **服务使用路径**：我的服务 → 智能体应用 → 添加服务
3. **服务上架路径**：我的应用 → 服务编辑 → 上架申请

### 7.3 移动端适配
- 响应式设计适配不同屏幕尺寸
- 触摸友好的按钮和交互元素
- 简化的移动端操作流程

## 8. 运营策略

### 8.1 内容运营
- **优质服务推荐**：编辑精选优质服务，提升首页推荐质量
- **开发者激励**：设立服务质量奖励，鼓励开发者提供优质服务
- **用户教育**：提供服务使用教程和最佳实践分享

### 8.2 社区建设
- **评价体系**：完善的用户评价和反馈机制
- **开发者社区**：提供开发者交流和技术支持平台
- **用户反馈**：建立用户需求收集和响应机制

## 9. 风险评估

### 9.1 技术风险
- **性能风险**：大量服务数据的查询和展示性能
- **兼容性风险**：不同类型服务的兼容性问题
- **安全风险**：服务权限控制和数据安全

### 9.2 业务风险
- **用户接受度**：新功能的用户学习成本
- **服务质量**：上架服务的质量控制
- **生态平衡**：服务提供者和使用者的利益平衡

### 9.3 风险应对
- 分阶段推出，先小范围试点
- 建立完善的服务审核机制
- 持续收集用户反馈并快速迭代
